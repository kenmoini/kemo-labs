# Mailcow - Full Email Stack

## Overview

Mailcow is a fully-featured, self-hosted email server suite that bundles Postfix (SMTP), Dovecot (IMAP/POP3), SOGo (webmail/calendaring/contacts), ClamAV (antivirus), Rspamd (spam filtering), and a web-based admin UI. It ships its own `docker-compose.yml` and manages ~15 containers internally. Mailcow uses its own bundled MariaDB and Redis instances and does NOT support using an external/shared database.

- **Project URL:** https://github.com/mailcow/mailcow-dockerized
- **Documentation:** https://docs.mailcow.email/
- **Static IP:** 192.168.62.80
- **DNS Zone:** lab.kemo.network

## Architecture Note - Mailcow's Own Compose Stack

Mailcow is opinionated and ships its own `docker-compose.yml`. It is designed to be cloned from GitHub and managed via its own `generate_config.sh` and `update.sh` scripts. Integrating it into a custom compose file is not recommended and not supported. The deployment approach should be:

1. Clone the mailcow repository into the deployment directory
2. Run `generate_config.sh` to produce `mailcow.conf`
3. Override settings in `mailcow.conf` and `docker-compose.override.yml`
4. Use `docker compose up -d` from within the mailcow directory

The wrapper compose or deployment script in this project should orchestrate calling mailcow's native tooling rather than redefining its services.

## Environment Variables (mailcow.conf)

Mailcow uses a `mailcow.conf` file (symlinked as `.env`) for all configuration. Key variables:

```bash
# Hostname - FQDN of the mail server (NOT the mail domain)
MAILCOW_HOSTNAME=mail.lab.kemo.network

# Timezone
TZ=America/New_York

# Database credentials (auto-generated by generate_config.sh)
DBNAME=mailcow
DBUSER=mailcow
DBPASS=<random-alphanumeric-28-chars>
DBROOT=<random-alphanumeric-28-chars>

# Redis
REDISPASS=<random-alphanumeric-28-chars>

# HTTP/S bindings - configure for Traefik integration
# To put web UI behind Traefik, bind to localhost or a specific port
HTTP_PORT=80
HTTP_BIND=192.168.62.80
HTTPS_PORT=443
HTTPS_BIND=192.168.62.80

# SMTP/IMAP bindings - bind to the static IP
SMTP_PORT=25
SMTPS_PORT=465
SUBMISSION_PORT=587
IMAP_PORT=143
IMAPS_PORT=993
POP_PORT=110
POPS_PORT=995
SIEVE_PORT=4190

# Internal ports (keep localhost-bound)
DOVEADM_PORT=127.0.0.1:19991
SQL_PORT=127.0.0.1:13306
REDIS_PORT=127.0.0.1:7654

# Project name
COMPOSE_PROJECT_NAME=mailcowdockerized

# Skip ClamAV if memory is tight (not needed with 128GB+)
SKIP_CLAMD=n

# Skip Let's Encrypt - we use StepCA via Traefik
SKIP_LETS_ENCRYPT=y

# Use custom ACME directory for StepCA (if using mailcow's built-in ACME)
# DIRECTORY_URL=https://stepca.lab.kemo.network/acme/acme/directory

# Additional SAN for certificates
ADDITIONAL_SAN=imap.*,smtp.*,autodiscover.*,autoconfig.*

# SOGo
SOGO_EXPIRE_SESSION=480

# Watchdog
USE_WATCHDOG=y

# Networking
IPV4_NETWORK=172.22.1
ENABLE_IPV6=false

# Password scheme
MAILCOW_PASS_SCHEME=BLF-CRYPT

# FTS (Full-Text Search)
SKIP_FTS=n
FTS_HEAP=512
FTS_PROCS=3
```

| `mysql-socket-vol-1` | MySQL socket (shared) | Negligible |
| `redis-vol-1` | Redis persistence | 100MB-1GB |

| `postfix-vol-1` | Postfix mail spool | 500MB-2GB |
| `postfix-tlspol-vol-1` | TLS policy cache | Negligible |
| `crypt-vol-1` | Mail encryption keys | Negligible |
| `sogo-web-vol-1` | SOGo web assets | 200MB |
| `sogo-userdata-backup-vol-1` | SOGo backups | 1-5GB |
| `clamd-db-vol-1` | ClamAV virus definitions | 500MB-1GB |

Additionally, Mailcow stores configuration and SSL data in bind mounts under `./data/`:
- `./data/conf/` - Configuration files for all services
- `./data/web/` - Web UI files
- `./data/assets/ssl/` - TLS certificates
- `./data/hooks/` - Lifecycle hooks

**Total minimum disk:** ~20GB for base install + email storage growth
**Recommended:** Dedicated partition/volume with 100GB+ for `/var/lib/docker/volumes/` or bind-mount vmail to a large disk.

## Network Configuration

### Static IP Binding

Mailcow needs its SMTP/IMAP ports bound directly to the host IP, NOT proxied through Traefik. SMTP protocol proxying through an HTTP reverse proxy is not possible.

```bash
# In mailcow.conf, bind mail ports to the static IP:
SMTP_PORT=25
SMTPS_PORT=465
SUBMISSION_PORT=587
IMAP_PORT=143
IMAPS_PORT=993
POP_PORT=110
POPS_PORT=995
SIEVE_PORT=4190
```

These ports bind to all interfaces by default. To restrict to the static IP, use `docker-compose.override.yml`:

```yaml
services:
  postfix-mailcow:
    ports:
      - "192.168.62.80:25:25"
      - "192.168.62.80:465:465"
      - "192.168.62.80:587:587"

  dovecot-mailcow:
    ports:
      - "192.168.62.80:143:143"
      - "192.168.62.80:993:993"
      - "192.168.62.80:110:110"
      - "192.168.62.80:995:995"
      - "192.168.62.80:4190:4190"

  nginx-mailcow:
    ports:
      - "192.168.62.80:80:80"
      - "192.168.62.80:443:443"
```

### Traefik Integration for Web UI

The Mailcow web UI (port 443) can optionally be put behind Traefik. Options:

1. **Direct access:** Access the web UI directly at `https://192.168.62.80` or `https://mail.lab.kemo.network` with mailcow managing its own TLS.
2. **Traefik proxy:** Disable mailcow's HTTPS binding, expose HTTP only on a high port, and let Traefik terminate TLS. This requires:
   - Setting `HTTPS_BIND=127.0.0.1` and `HTTP_BIND=0.0.0.0` with a non-standard HTTP_PORT
   - Setting `SKIP_LETS_ENCRYPT=y` (Traefik/StepCA handles certs)
   - Adding Traefik labels to nginx-mailcow in `docker-compose.override.yml`

**Recommendation:** Let Mailcow handle its own web TLS directly, skip Traefik for this service. Use StepCA via mailcow's ACME client (`DIRECTORY_URL`) or manually provision certs.

### Internal Network

Mailcow creates its own Docker bridge network (`br-mailcow`) with subnet `172.22.1.0/24`. This is isolated from other Docker services. The `netfilter-mailcow` container runs in host network mode with privileged access for iptables-based ban management.

### Macvlan / Host Network Considerations

Since Mailcow needs many ports on a specific IP, consider:
- **Option A (simpler):** Assign 192.168.62.80 as a secondary IP on the host and bind mailcow ports to it via `docker-compose.override.yml` port mappings.
- **Option B (macvlan):** Use a macvlan network to give the mailcow stack its own IP. This is more complex and may conflict with mailcow's internal networking.

**Recommendation:** Option A -- add 192.168.62.80 as a secondary IP on the host's LAN interface.

## DNS Requirements

The following DNS records must be configured in the `lab.kemo.network` zone:

```
; A record for mail server
mail.lab.kemo.network.          IN A     192.168.62.80

; MX record for the domain
lab.kemo.network.               IN MX    10 mail.lab.kemo.network.

; Autodiscover/Autoconfig for email clients
autodiscover.lab.kemo.network.  IN CNAME mail.lab.kemo.network.
autoconfig.lab.kemo.network.    IN CNAME mail.lab.kemo.network.

; SPF record
lab.kemo.network.               IN TXT   "v=spf1 mx a:mail.lab.kemo.network -all"

; DKIM record (generated by Mailcow after first run)
; dkim._domainkey.lab.kemo.network. IN TXT "v=DKIM1; k=rsa; p=..."

; DMARC record
_dmarc.lab.kemo.network.        IN TXT   "v=DMARC1; p=quarantine; rua=mailto:postmaster@lab.kemo.network"

; Reverse DNS (PTR) - must be set in ISP/hosting panel
; 80.62.168.192.in-addr.arpa.   IN PTR   mail.lab.kemo.network.
```

## Special Considerations

### 1. Mailcow Uses Its Own Compose - Do Not Merge

Mailcow should be deployed from its own cloned repository. Do not attempt to merge its services into a project-wide `docker-compose.yml`. Use `docker-compose.override.yml` for customizations.

### 2. TLS Certificate Strategy

Since the lab uses StepCA for ACME:
- Set `SKIP_LETS_ENCRYPT=y` in `mailcow.conf`
- Either set `DIRECTORY_URL` to the StepCA ACME endpoint for mailcow's built-in ACME client, OR manually provision and mount certificates to `./data/assets/ssl/`
- The cert must cover: `mail.lab.kemo.network`, `autodiscover.lab.kemo.network`, `autoconfig.lab.kemo.network`

### 3. ClamAV Startup Time

ClamAV takes 2-5 minutes to load virus definitions into memory on first start. During this time, email scanning is degraded. The watchdog container monitors this and will report when ClamAV is ready.

### 4. Firewall / Port Conflicts

Ensure no other services on the host are using ports 25, 110, 143, 465, 587, 993, 995, or 4190. Common conflicts:
- Fedora's built-in Postfix (disable `postfix.service`)
- Cockpit or other web UIs on port 443

### 5. Netfilter Container Requires Privileged Mode

The `netfilter-mailcow` container runs in `privileged` mode and `network_mode: host` to manage iptables rules for IP banning. This is a security consideration.

### 6. Docker Socket Exposure

Both `dockerapi-mailcow` and `ofelia-mailcow` mount `/var/run/docker.sock` read-only. This is required for container management and scheduled tasks.

### 7. SELinux on Fedora

Mailcow's compose file uses `:Z` and `:z` volume mount suffixes for SELinux labeling. This should work correctly on Fedora, but if issues arise, check `audit.log` for AVC denials.

### 8. Backup Strategy

Key data to back up:
- `./data/` directory (all configuration)
- `vmail-vol-1` Docker volume (all email data)
- `mysql-vol-1` Docker volume (database) -- or use `mysqldump` via `docker compose exec mysql-mailcow mysqldump --all-databases`
- `crypt-vol-1` (encryption keys -- CRITICAL, loss means encrypted mail is unrecoverable)
